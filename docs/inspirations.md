# SSUBB Inspirations

这份文档不是竞品分析，也不是“功能越多越好”的对标清单。

它的目的很简单：

- 记录 `VideoCaptioner` 和 `subgen` 里真正值得 SSUBB 借鉴的地方
- 明确哪些东西适合学，哪些东西不该照搬
- 给后续开发一个判断标准，避免项目越做越重、越做越拧巴

这份文档最重要的前提是：

> SSUBB 的初衷仍然是“我一个人长期自用的异地字幕系统”。
> 在保障功能性的前提下，`简单`、`符合直觉`、`优雅自然` 是第一原则。

---

## 1. SSUBB 的核心原则

后续所有“偷师”和演进，都建议遵守这几条原则。

### 1.1 单人自用优先

SSUBB 不是先面向大众用户设计，再回头适配你自己的环境。

它应该首先服务于你自己的真实使用方式：

- 家里 NAS 负责接入和写回
- 异地 GPU 主机负责计算
- 配置和维护者基本只有你自己

这意味着很多“通用平台能力”并不是越早做越好。

### 1.2 简单优先于大而全

只要当前问题能用更小、更稳、更直观的方式解决，就不要急着引入复杂抽象。

比如：

- 能用单 Worker 跑稳，就不要过早做多 Worker 调度
- 能用清晰配置文件解决，就不要过早做复杂数据库配置中心
- 能用一个好用的启动器解决，就不要过早搞完整桌面平台

### 1.3 直觉优先于“高级”

对你未来自己来说，最好的系统不是“功能最多”，而是“半年后回来还看得懂、改得动”。

所以更推荐：

- 明确的目录结构
- 少量但清晰的配置项
- 符合直觉的任务状态
- 尽量少的隐藏行为

### 1.4 优雅来自边界清晰

SSUBB 后面如果要继续长大，最重要的不是把功能一直加，而是把边界守住：

- Coordinator 负责接入、编排、写回
- Worker 负责计算
- Launcher 负责启动、环境检测、模型管理

只要边界清晰，系统自然会显得优雅。

## 2. 从 VideoCaptioner 可以偷师什么

`VideoCaptioner` 最强的地方，不是它功能多，而是它把“工具入口”和“配置体验”做得很顺。

对 SSUBB 来说，最值得借的不是整套产品形态，而是这些思路。

### 2.1 配置优先级设计

`VideoCaptioner` 的配置优先级非常清楚：

- 命令行参数
- 环境变量
- 用户配置文件
- 默认值

这套设计很适合 SSUBB 尤其是未来的 Worker 客户端。

SSUBB 可以借鉴成：

- 启动器/CLI 参数
- `SSUBB_*` 环境变量
- `config.yaml` 或用户级配置文件
- 默认值

为什么值得学：

- 行为可预测
- 排查问题时更容易
- 后面做 launcher 也不用推翻当前配置体系

### 2.2 CLI 命令分层

`VideoCaptioner` 很擅长把不同能力拆成清晰的命令入口。

SSUBB 后续很适合偷这个思路，但不需要一上来做很大。

可以先只考虑 Worker 这一侧，未来逐步补下面这些命令：

- `ssubb-worker run`
- `ssubb-worker doctor`
- `ssubb-worker config`
- `ssubb-worker model`
- `ssubb-worker test`

为什么值得学：

- 便于本地调试
- 便于启动器调用
- 便于后续做 GUI 外壳

### 2.3 配置管理子命令

`VideoCaptioner` 的 `config show / get / set / init / edit / path` 很适合借鉴。

SSUBB 后面如果做 Worker 客户端，完全可以复用这个思路：

- `config init` 初始化配置
- `config show` 看当前配置
- `config path` 找到配置文件
- `config set` 修改某个关键项

为什么值得学：

- 比直接让自己手改 YAML 更友好
- 对以后做 launcher 很有帮助

### 2.4 GUI / CLI 双入口

`VideoCaptioner` 一个很成熟的思路是：

- GUI 给普通使用
- CLI 给高级使用和自动化
- 两者共享同一套内核能力

这对 SSUBB 的启发很大。

SSUBB 后面最自然的形态其实不是“只有脚本”或者“只有桌面应用”，而是：

- Coordinator 继续保留服务端形态
- Worker 逐步具备 CLI + 启动器双入口

### 2.5 默认能力开箱即用

`VideoCaptioner` 非常重视“没有复杂配置也能先用起来”。

这点很值得 SSUBB 学。

对 SSUBB 来说可以转化为：

- 没有 LLM Key 时，至少能稳定转写
- 有 LLM Key 时，再开启优化和翻译增强
- 没有高级配置时，也能跑通默认链路

为什么值得学：

- 降低维护心智负担
- 让系统更符合“自然可用”的感觉

### 2.6 输出策略意识

`VideoCaptioner` 对输出格式、布局、风格、模式的意识很强。

SSUBB 不需要一上来追求 UI 花样，但可以吸收这些思路：

- 保留原始转写字幕
- 保留优化后字幕
- 保留翻译后字幕
- 支持双语字幕输出
- 支持更清晰的命名格式

这不是为了“功能多”，而是为了让后续结果更可控。

## 3. 从 subgen 可以偷师什么

`subgen` 最值得借的不是界面，而是“长期挂着跑”的经验。

SSUBB 如果后面真想长期稳定自用，很多细节都应该向 `subgen` 学。

### 3.1 环境变量与服务配置治理

`subgen` 的环境变量虽然很多，但分类非常清楚。

这对 SSUBB 的启发不是“也去搞一堆变量”，而是：

- 要把配置分层
- 要把配置分类
- 要让配置名字一眼能看懂

建议 SSUBB 后续保持这几组配置边界：

- 核心服务配置
- Worker 计算配置
- 字幕策略配置
- 集成配置
- 路径映射配置

### 3.2 跳过逻辑

`subgen` 很强的一点是“尽量别白跑”。

它会做很多跳过判断，比如：

- 已有目标字幕时跳过
- 有内封字幕时跳过
- 指定语言音轨不需要处理时跳过
- 不符合偏好语言时跳过

这非常适合 SSUBB。

因为 SSUBB 是长期自动化系统，不是一次性处理工具。

对 SSUBB 来说，值得借鉴的方向有：

- 更细的已有字幕判断
- 更细的目标语言判断
- 更细的内封字幕和外置字幕判断
- 更细的音轨选择与跳过策略

### 3.3 路径映射与媒体服务器一致性

`subgen` 非常重视一个原则：

> 媒体服务器看到的路径，处理服务最好也能看到同样的路径。

这个原则和 SSUBB 高度一致。

SSUBB 后续值得直接加强：

- 路径映射规则文档化
- 启动前路径校验
- WebUI 中增加路径诊断
- MoviePilot / Emby 路径映射自检

### 3.4 触发器设计

`subgen` 支持很多触发方式：

- webhook
- 播放触发
- 新增媒体触发
- 扫描目录
- 监控目录

SSUBB 不需要全搬，但可以吸收成几个自然的触发模式：

- MoviePilot 自动触发
- 手动路径触发
- 扫描目录补处理
- 监控“待处理目录”

这里要注意：

- 不要为了“强大”把入口做得过多
- 保持 2 到 4 种最自然的触发方式就够了

### 3.5 长期运行细节

`subgen` 很多成熟度其实都来自这些细节：

- 转写超时
- 队列控制
- 显存清理
- 模型清理延迟
- 任务完成回调
- rootless / 权限适配

这些不是大功能，但非常值得 SSUBB 借鉴。

对 SSUBB 来说，优先级很高的细节包括：

- 任务阶段超时
- Worker 资源清理
- 任务完成 / 失败事件回调
- 权限和路径写回失败诊断

### 3.6 音轨与语言处理

`subgen` 在这方面做得很细，尤其适合媒体库场景。

SSUBB 后续可以吸收这些方向：

- 首选音轨语言
- 多音轨选择策略
- 错误语言强制修正
- 检测语言时跳过片头片尾
- 只处理某些语言

这类能力特别适合自动化系统，因为它们能显著减少错跑和误跑。

## 4. 哪些东西不该直接照搬

借鉴不等于复制。

有些东西虽然它们有，但 SSUBB 不应该急着做。

### 4.1 不要过早堆过多开关

`subgen` 的配置很强，但也很容易变复杂。

SSUBB 如果照搬大量 ENV 和开关，很容易失去“自用系统”的简洁感。

建议：

- 先只保留最常用、最关键的配置项
- 高级项放到二级配置或高级模式

### 4.2 不要过早追求全平台万能

`VideoCaptioner` 和 `subgen` 都在努力兼容很多场景。

但 SSUBB 当前不需要先追求：

- 所有平台都同样完美
- 所有媒体服务器都无缝兼容
- 所有模型后端都支持

当前最自然的边界仍然应该是：

- Coordinator：NAS / Docker
- Worker：Windows / NVIDIA GPU

先把这条线做优雅，再扩展。

### 4.3 不要为了产品化而牺牲边界

如果以后做 Worker launcher，也不要把所有逻辑塞回 Worker 主进程里。

应保持：

- Launcher 是 launcher
- Worker 是 worker
- Coordinator 是 coordinator

这比“表面上一体化”更优雅。

## 5. 对 SSUBB 最有价值的偷师结论

如果把两边的优点抽象成 SSUBB 自己的路线，可以概括成三句话：

### 5.1 学 VideoCaptioner 做 Worker 的“入口体验”

重点学习：

- 配置优先级
- CLI / GUI 双入口
- 启动器思路
- 默认配置可用性

### 5.2 学 subgen 做 Coordinator 的“长期运行能力”

重点学习：

- 跳过逻辑
- 媒体服务器集成
- 路径映射
- 资源清理与超时控制
- 完成回调与自动化链路

### 5.3 保留 SSUBB 自己的独特定位

SSUBB 真正不同的地方不是某个单点功能，而是这条结构路线：

- 家里轻节点负责接入和写回
- 异地强节点负责算力
- 两端松耦合协作

这个定位本身就很有价值，不需要为了“像别的项目”而抹掉自己的特性。

---

## 6. 建议转化成的后续任务

基于这次偷师分析，后续最值得落成任务的方向有：

### 6.1 借鉴 VideoCaptioner 的任务

- 设计 SSUBB Worker 的 CLI 体系
- 设计统一配置优先级
- 设计 Worker 启动器 / launcher 的交互流程
- 设计默认可用配置与高级配置分层

### 6.2 借鉴 subgen 的任务

- 重构字幕跳过逻辑
- 增强音轨选择与语言判断
- 增加路径映射校验
- 增加任务超时与资源清理
- 增加任务完成 / 失败事件回调

### 6.3 保持 SSUBB 简洁性的约束任务

- 定义“必须配置项最小集合”
- 定义“默认路径”
- 定义“单 Worker 优先”的设计边界
- 定义“不做清单”，避免项目越来越重

---

## 7. 最后判断

对 SSUBB 来说，最好的“偷师”不是把 `VideoCaptioner` 和 `subgen` 各自的功能全搬过来。

最好的做法是：

- 用 `VideoCaptioner` 的思路，让 Worker 更像一个自然好用的客户端
- 用 `subgen` 的经验，让 Coordinator 更像一个稳妥耐跑的服务
- 用 SSUBB 自己的定位，把两者组合成一个适合你长期自用的异地字幕系统

如果后面每次新增功能前都先问一句：

> 这个改动是让系统更简单、更直觉、更优雅，还是只是让它“看起来更强”？

那 SSUBB 大概率会越做越顺，而不是越做越重。
