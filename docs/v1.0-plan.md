# SSUBB V1.0 详细规划：开箱即用

## 愿景

让 SSUBB 从"开发者工具"变成"普通用户产品"。下载即用，双击启动，无需接触命令行、Python 环境或 YAML 配置。

**核心指标**：从下载到处理第一个视频，全程 < 5 分钟，零命令行操作。

---

## Phase 1：跨平台打包

### 目标
将 Coordinator 和 Worker 打包为独立可执行文件，消除 Python 环境依赖。

### 技术选型

| 方案 | 优点 | 缺点 | 推荐 |
|------|------|------|------|
| PyInstaller | 成熟稳定，社区大 | 启动慢，包体大 | Worker 端 |
| Nuitka | 启动快，可编译为原生 | 构建复杂，调试困难 | Coordinator 端 |
| cx_Freeze | 跨平台一致 | 社区较小 | 备选 |

**决策**：Coordinator 用 Nuitka（启动速度优先，NAS 资源有限），Worker 用 PyInstaller（GPU 依赖链复杂，PyInstaller 的 hook 机制更成熟）。

### 打包清单

**Coordinator 包**：
```
ssubb-coordinator-{version}-win64.zip
ssubb-coordinator-{version}-linux-x64.tar.gz
├── ssubb-coordinator.exe (或 ssubb-coordinator)
├── data/                    # 首次运行自动创建
├── config.yaml              # 首次运行自动生成
└── README.txt
```

**Worker 包**：
```
ssubb-worker-{version}-win64-cuda12.zip
ssubb-worker-{version}-linux-x64-cuda12.tar.gz
├── ssubb-worker.exe (或 ssubb-worker)
├── models/                  # Whisper 模型缓存
├── data/
├── config.yaml
└── README.txt
```

### 关键挑战

1. **FFmpeg 内嵌**：Coordinator 依赖 FFmpeg 抽音频。方案：打包 `ffmpeg-static` 二进制，运行时自动解压到临时目录。
2. **Whisper 模型分发**：模型文件 1-3GB，不适合打包进二进制。方案：首次启动时自动下载到 `models/` 目录，支持断点续传。
3. **CUDA 依赖**：Worker 需要 CUDA runtime。方案：提供 `cuda11` / `cuda12` 两个变体包，或动态检测系统 CUDA 版本。
4. **跨平台路径**：Windows `\` vs Linux `/`。方案：统一使用 `pathlib.Path`，配置文件使用相对路径。

### 产出物

- `scripts/build_coordinator.py` — Nuitka 构建脚本
- `scripts/build_worker.py` — PyInstaller 构建脚本
- `.github/workflows/release.yml` — CI 自动构建 + GitHub Release

## Phase 2：桌面启动器

### 目标
提供原生 GUI 启动器，替代命令行操作。

### 技术选型

| 方案 | 优点 | 缺点 | 推荐 |
|------|------|------|------|
| PySide6 (Qt) | 原生感强，跨平台 | 包体大 (~50MB) | Worker 端 |
| Tkinter | 零依赖，Python 内置 | 丑，功能有限 | 备选 |
| Tauri (Rust+Web) | 包体小，Web 技术栈 | 需要 Rust 工具链 | 备选 |
| Electron | Web 技术栈 | 包体巨大 (~150MB) | 不推荐 |

**决策**：使用 PySide6，与现有 Python 代码栈一致，避免引入新语言。

### Coordinator 端启动器

Coordinator 通常运行在 NAS (Linux) 或 Windows 桌面。两种场景：

**NAS (Docker)**：不需要桌面启动器，继续使用 Docker Compose + WebUI 引导。

**Windows 桌面**：提供轻量启动器，功能简化：
- 启动/停止 Coordinator 服务
- 打开浏览器访问 WebUI
- 查看基本状态（端口、Worker 数量）
- 系统托盘常驻

## Phase 3：首次运行体验 (OOBE)

### 目标
首次启动时自动检测环境、引导配置、下载模型，全程无命令行。

### 流程

```
首次启动
    │
    ├─ 1. 环境检测（自动）
    │   ├─ Python 版本 ✅
    │   ├─ CUDA 版本 ✅ → 记录 GPU 型号
    │   ├─ FFmpeg ✅ → 内嵌 or 提示安装
    │   └─ 网络连通性 ✅ → 测试 Coordinator 地址
    │
    ├─ 2. 模型下载（自动 + 进度条）
    │   ├─ Whisper 模型选择 (base/small/medium/large-v3-turbo)
    │   ├─ 断点续传下载
    │   └─ 校验 SHA256
    │
    ├─ 3. 配置引导（交互式）
    │   ├─ Coordinator 地址
    │   ├─ Worker ID
    │   ├─ LLM API Key
    │   ├─ 翻译目标语言
    │   └─ 保存 config.yaml
    │
    └─ 4. 启动服务
        ├─ 启动 Worker
        ├─ 心跳注册
        └─ 进入主界面
```

### 环境检测详细方案

| 检测项 | 方法 | 失败处理 |
|--------|------|----------|
| Python 版本 | `sys.version` | 不应发生（打包内嵌） |
| CUDA | `nvidia-smi` + `torch.cuda.is_available()` | 提示安装驱动，提供下载链接 |
| FFmpeg | `subprocess.run(['ffmpeg', '-version'])` | 使用内嵌版本 |
| Whisper 模型 | 检查 `models/` 目录 | 自动下载 |
| Coordinator 连通 | `http.get(coordinator_url + '/api/status')` | 提示检查地址/网络 |
| 磁盘空间 | `shutil.disk_usage()` | < 2GB 时警告 |

## Phase 5：自动更新

### 目标
启动时检查新版本，一键更新。

### 方案

1. **检查更新**：启动时请求 GitHub Releases API (`/repos/SSUBB/SSUBB/releases/latest`)
2. **版本比较**：语义化版本比较 (`0.12.0` < `1.0.0`)
3. **下载更新**：下载对应平台的安装包到临时目录
4. **应用更新**：
   - Windows：替换 exe + 重启
   - Linux：替换二进制 + 重启
   - Docker：提示 `docker pull` 最新镜像
5. **回滚机制**：更新前备份当前版本到 `backup/`，失败时可回滚

### 更新流程

```
启动 → 检查更新 → 发现新版本 v1.1.0
    │
    ├─ 弹窗: "发现新版本 v1.1.0，更新内容: ..."
    │
    ├─ [立即更新] → 下载 (进度条) → 替换 → 重启
    │
    └─ [稍后提醒] → 跳过，下次启动再检查
```

## Phase 6：系统托盘 & 后台运行

### 目标
最小化到系统托盘，后台常驻，不影响日常使用。

### 功能

- **托盘图标**：运行中 = 绿色图标，停止 = 灰色图标，有任务 = 闪烁
- **右键菜单**：
  - 打开主窗口
  - 启动 / 停止服务
  - 查看当前任务数
  - 打开 WebUI (Coordinator)
  - 退出
- **通知气泡**：任务完成/失败时弹出系统通知
- **开机自启**：可选，注册为系统启动项

## 构建与发布

### CI/CD 流程

```yaml
# .github/workflows/release.yml
on:
  push:
    tags: ['v*']

jobs:
  build-coordinator:
    strategy:
      matrix:
        os: [windows-latest, ubuntu-latest]
    steps:
      - uses: actions/checkout@v4
      - name: Build with Nuitka
        run: python scripts/build_coordinator.py
      - name: Upload artifact
        uses: actions/upload-artifact@v4

  build-worker:
    strategy:
      matrix:
        os: [windows-latest, ubuntu-latest]
        cuda: [cuda11, cuda12]
    steps:
      - uses: actions/checkout@v4
      - name: Build with PyInstaller
        run: python scripts/build_worker.py
      - name: Upload artifact
        uses: actions/upload-artifact@v4

  release:
    needs: [build-coordinator, build-worker]
    runs-on: ubuntu-latest
    steps:
      - name: Create GitHub Release
        uses: softprops/action-gh-release
        with:
          files: artifacts/*
```

### 发布产物

```
GitHub Release v1.0.0
├── ssubb-coordinator-1.0.0-win64.zip          (35 MB)
├── ssubb-coordinator-1.0.0-linux-x64.tar.gz   (30 MB)
├── ssubb-worker-1.0.0-win64-cuda12.zip        (500 MB)
├── ssubb-worker-1.0.0-win64-cuda11.zip        (500 MB)
├── ssubb-worker-1.0.0-linux-x64-cuda12.tar.gz (480 MB)
```

## 与现有架构的关系

V1.0 不改变 SSUBB 的核心架构（Coordinator-Worker 分离），只改变**交付方式**：

```
V0.12 (当前)                    V1.0 (目标)
┌─────────────┐                ┌─────────────────┐
│ pip install  │                │ 下载 zip        │
│ python -m    │      →        │ 双击 exe        │
│ edit yaml    │                │ GUI 引导配置    │
│ 命令行启动   │                │ 系统托盘常驻    │
└─────────────┘                └─────────────────┘
```

WebUI 控制台仍然是主要的管理和监控界面，桌面启动器只是替代了命令行启动和环境管理的部分。

## 风险与缓解

| 风险 | 影响 | 缓解 |
|------|------|------|
| CUDA 版本碎片化 | Worker 包需要适配多个 CUDA 版本 | 提供 cuda11/cuda12 两个变体；首次启动检测并提示 |
| 包体过大 | Worker 包 > 500MB，下载慢 | 模型/CUDA 可选下载；提供 lite 版本 |
| Windows Defender 误报 | PyInstaller 打包的 exe 常被误报 | 代码签名证书；提交白名单申请 |
| 跨平台 UI 一致性 | PySide6 在不同平台渲染差异 | 使用 QSS 统一样式，避免原生控件 |
| 自动更新安全性 | 中间人攻击替换二进制 | SHA256 校验 + HTTPS + GitHub 签名 |
