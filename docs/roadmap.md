# SSUBB Roadmap

这份文档用于沉淀 SSUBB 接下来一段时间的完整开发计划，目标不是只列几个模糊方向，而是提供一套可持续执行的开发底稿。

适用场景：

- 继续个人长期开发时，随时回看项目当前阶段和下一步重点
- 拆分 issue、milestone、周计划时作为母版
- 在不重读全部源码的情况下，快速理解项目要往哪里迭代

---

## 0. 最小可用上线版本规划

在进入完整 roadmap 之前，先明确一个最重要的原则：

> SSUBB 不是先做成“大而全的平台”，再慢慢找使用场景。
> 它应该先尽快实现一个最小可用版本，稳定解决“更方便看没有字幕的电影电视”这个核心问题。

所以接下来的版本规划，建议遵循这个顺序：

1. 先做出能长期自用的最小可用版本
2. 再补工程化和稳定性
3. 再补轻量自动化
4. 最后才做 launcher / exe / 多 Worker 这类高级优化

### V0.1 最小可用上线版 ✅ (已完成 2026-04)

目标：

- 先把“补字幕”这件事真正跑起来
- 不追求花哨体验
- 不追求多节点复杂能力
- 只要能稳定帮助你看没字幕或字幕差的影视内容，就算成功

V0.1 应具备的功能：

- Coordinator / Worker 双节点主链路可用
- NAS 侧可提交任务、抽音频、接收结果、写回字幕
- Worker 侧可完成转写、基础翻译、基础优化
- 支持手动提交任务
- 支持至少一种自动触发方式
  - 推荐优先支持 MoviePilot 入库触发
- 支持字幕存在性检查与基本跳过逻辑
- 支持任务状态查看
- 支持日志查看
- 支持端到端跑通从媒体文件到 `.srt` 写回

V0.1 明确不要求：

- 多 Worker
- launcher / exe 客户端
- 很完整的 UI
- 很复杂的调度系统
- 很多高级字幕策略

V0.1 的验收标准：

- 你能稳定地把“没有字幕的电影/电视剧”交给 SSUBB 处理
- 处理完成后字幕能正确回到媒体目录
- 日常使用不需要频繁改代码和手工介入

### V0.2 稳定可维护版 ✅ (已完成 2026-04)

目标：

- 在 V0.1 可用的基础上，把它变成更省心的自用系统

V0.2 应补的功能：

- `config.example.yaml`
- 配置项梳理与文档化
- 任务状态机重构
- 错误分类
- 重试与恢复机制
- 更可靠的任务详情和失败原因展示
- 更稳定的路径映射和部署说明
- smoke test / 基础集成测试

V0.2 的关键词：

- 稳定
- 可维护
- 少踩坑

### V0.3 轻量自动化版 ✅ (已完成 2026-04)

目标：

- 让“补字幕”更自然地在后台发生

V0.3 应补的功能：

- 夜间扫描新增媒体
- Worker 空闲时补处理缺字幕内容
- 剧集下一集预热
- 与实时触发链路去重
- 轻量配置开关

V0.3 的关键词：

- 自然
- 安静
- 不打扰

### V0.4 体验增强版 ✅ (已完成 2026-04)

目标：

- 在不破坏简洁性的前提下，让使用体验更顺

V0.4 应补的功能：

- 更好的 WebUI
- 更清晰的任务详情页
- 更好的 Worker 状态展示
- 更丰富的字幕输出策略
- 更细的跳过逻辑和音轨选择

V0.4 的关键词：

- 顺手
- 清晰
- 优雅

### V0.5 高级优化版 ✅ (已完成 2026-04, exe 封装延后)

目标：

- 做那些“不是必须，但会显著提升长期体验”的高级能力

V0.5 可以考虑的功能：

- Worker launcher / exe 客户端
- 模型下载与环境检测
- 更低门槛的异地节点迁移
- 更完整的运维面板

这一版开始，才适合认真推进：

- launcher
- exe 封装
- 更强产品化入口

### V0.6+ 扩展能力版

目标：

- 在前面所有能力足够稳定后，再考虑更复杂的系统能力

V0.6+ 可以考虑的功能：

- 多 Worker
- 调度策略
- 节点管理
- 更复杂的任务路由

结论：

- `V0.1 ~ V0.3` 才是当前真正应该优先做的主线
- `V0.5+` 才是 launcher / exe / 更重产品化的合理阶段
- 任何会拖慢 V0.1 上线的高级能力，都应该主动后置

## 1. 总体目标

SSUBB 的长期目标不是单纯“做字幕”，而是演进成一套适合家庭媒体场景的分布式字幕基础设施：

- 家里的 NAS 或轻节点作为接入与编排中心
- 异地 GPU 机器作为算力节点
- 两端通过稳定协议协作
- 后续支持更低门槛部署、更强恢复能力、更完整的多节点能力

它最核心、最朴素的目标始终不变：

- 让我更方便地看没有字幕、字幕质量差或者字幕不同步的电影和电视剧
- 尽量把“补字幕”这件事变成后台自然完成的动作，而不是额外工作

从工程推进上，建议分成 4 个阶段：

1. `P0` 稳定现有双节点主链路
2. `P1` 补足工程化能力
3. `P2` 打磨成高可用的单 Worker 产品
4. `P3` 推进 Worker 客户端化和多节点演进

---

## 2. 当前阶段判断

基于当前仓库状态，可以把项目定位为：

- 主链路可运行的工程原型
- 单 Worker 架构已成立
- 适合继续迭代，不适合直接对外作为“零门槛成品”发布

当前最重要的原则：

- 先把现有链路做稳
- 再做体验和工程升级
- 最后再做客户端化、多 Worker 和产品化扩张
- 不为了“平台感”而把一个自用系统做得过重

---

## 3. 分阶段路线

### P0：稳定现有双节点主链路

目标：

- 确保 NAS -> Worker -> 字幕写回这条链路足够稳定
- 明确当前版本能做什么、不能做什么
- 把现有部署和运行方式固化下来

#### Task P0-1：梳理当前真实能力边界

任务说明：

- 输出“当前已支持功能”
- 输出“当前明确不支持功能”
- 清理 README / 架构文档中偏超前的说法
- 明确当前只承诺单 Worker 运行模式

详细子任务：

- 梳理 Coordinator 当前已提供的 API 和行为
- 梳理 Worker 当前已支持的执行链路
- 梳理 MoviePilot 插件当前已实现的自动触发与回调
- 标记哪些能力是“代码里已完成”，哪些只是“规划中”
- 对 README、architecture 文档做一次真实能力校对

验收标准：

- 文档描述和代码行为一致
- 不再出现“文档像产品、代码像原型”的错位

#### Task P0-2：跑通标准部署链路

任务说明：

- 固化当前可用的 Coordinator 部署方式
- 固化当前可用的 Worker 部署方式
- 明确最小配置项与必要环境

详细子任务：

- 确认 Coordinator 在 Docker 下的标准启动方式
- 确认 Worker 在 Windows 原生环境下的标准启动方式
- 验证 `config.yaml` 中哪些字段是必须的
- 明确媒体路径映射约束
- 至少验证一条完整任务链路

验收标准：

- 新环境按文档能完成部署
- 能成功触发任务并生成字幕

#### Task P0-3：定义当前版本基线

任务说明：

- 给当前可用状态定义一个里程碑版本
- 定义当前版本的明确边界

建议基线：

- 支持单 Worker
- 支持音频提取、转写、翻译、字幕写回
- 支持基础 WebUI
- 支持 MoviePilot 插件雏形
- 支持围绕“补看无字幕影视内容”这一核心场景稳定工作

暂不承诺：

- 多 Worker 调度
- 完整自动恢复
- 稳定客户端化分发

验收标准：

- 项目有清晰“当前版”的定义
- 后续开发不再围绕模糊目标摇摆

### P1：补足工程化能力

目标：

- 把“能跑”提升到“能维护”
- 让项目具备持续演进基础

#### Task P1-1：配置体系重构

任务说明：

- 建立完整、清晰、脱敏的配置体系

详细子任务：

- 新增 `config.example.yaml`
- 梳理现有全部配置项
- 标记哪些字段当前真实被使用
- 删除未用字段，或明确标记为预留字段
- 区分 Coordinator / Worker / 集成配置 / 模型配置
- 明确环境变量覆盖优先级
- 把敏感字段从实际示例中移除
- 增加“最小可运行配置示例”
- 增加“推荐生产配置示例”

建议产出：

- `config.example.yaml`
- `docs/configuration.md`

验收标准：

- 不看源码也能完成配置
- 新用户知道哪些必须填，哪些可以默认

#### Task P1-2：部署与启动标准化

任务说明：

- 把当前部署流程标准化成可重复步骤

详细子任务：

- 固化 Coordinator 的 Docker 部署
- 固化 Worker 的原生部署
- 增加环境前置条件说明
- 增加部署检查清单
- 考虑增加启动脚本与环境检查脚本
- 确认所有命令都能从仓库根目录执行

建议新增文件：

- `scripts/check_env.ps1`
- `scripts/start_worker.ps1`
- `scripts/start_coordinator.ps1`
- `docs/deployment.md`

验收标准：

- 新机器可以按步骤快速部署
- 部署文档足够短且明确

#### Task P1-3：任务状态机重构

任务说明：

- 把当前任务执行过程重构为更清晰的阶段化状态机

建议状态：

- `created`
- `subtitle_checking`
- `extracting_audio`
- `uploading`
- `worker_queued`
- `transcribing`
- `optimizing`
- `translating`
- `writing_subtitle`
- `refreshing_emby`
- `completed`
- `failed`
- `skipped`

详细子任务：

- 重构现有 `TaskStatus`
- 给每个状态定义进入条件、退出条件、超时规则
- 设计状态迁移图
- 在数据库中补阶段信息
- 兼容旧状态字段或做迁移策略

验收标准：

- 任何任务都能准确定位到具体阶段
- 状态流转更适合 UI 展示和自动恢复

#### Task P1-4：错误分类与失败治理

任务说明：

- 不再把所有异常都收敛成一个“失败”

建议错误分类：

- 配置错误
- 网络错误
- 媒体读取错误
- 音频提取错误
- 上传错误
- 模型错误
- LLM 错误
- 写回字幕错误
- Emby 刷新错误
- 回调错误

详细子任务：

- 为任务表增加 `error_code`
- 记录失败阶段
- 记录最后错误消息
- 区分可自动重试错误与人工介入错误
- 在 WebUI 中展示更友好的失败原因

验收标准：

- 用户看到失败时能知道大概问题在哪

#### Task P1-5：恢复与重试机制

任务说明：

- 提高系统面对断网、重启、卡死时的自愈能力

详细子任务：

- 设计阶段超时机制
- 设计上传中断恢复
- 设计 Worker 掉线恢复
- 设计回调失败补偿
- 设计最大重试次数和退避策略
- 区分“安全重试”和“不可安全重试”

验收标准：

- 常见故障不会把任务永久卡在未知状态

#### Task P1-6：日志与可观测性

任务说明：

- 建立最基本的可观测性体系

详细子任务：

- 统一日志格式
- 日志中强制带 `task_id`
- 记录阶段耗时
- 区分系统日志与任务日志
- 统计成功率、失败率、平均耗时
- 为后续 metrics 接口预留结构

建议产出：

- `docs/observability.md`

验收标准：

- 出现问题时能较快定位原因

验收标准：

#### Task P2-4：MoviePilot / Emby 集成增强

任务说明：

验收标准：

- 集成体验更稳定、更接近自动化生产线

#### Task P2-5：定时任务与闲时任务

任务说明：

- 增加轻量级的自动补字幕能力，更贴合个人自用场景

设计目标：

- 不把它做成复杂的通用调度系统
- 只解决最自然的几个问题：
  - 夜间自动扫描新增媒体
  - Worker 空闲时补处理缺字幕内容
  - 连续剧预热下一集字幕

详细子任务：

- 定义“定时扫描”模式
- 定义“Worker 空闲时触发”模式
- 定义“最近新增媒体补处理”模式
- 定义“下一集预热”模式
- 增加基本的去重与节流逻辑
- 避免与实时触发链路重复创建任务
- 在配置中增加简单直观的开关，而不是大量复杂 cron 规则

建议配置形态：

- 是否开启夜间扫描
- 夜间扫描时间窗口
- 是否只在 Worker 空闲时运行
- 是否对剧集启用下一集预热
- 每次最多补处理多少个任务

验收标准：

- 用户不必每次手动点任务
- 系统可以在“适合的时候”自然补齐字幕
- 配置仍然保持简单，符合自用项目的直觉

### P3：Worker 客户端化与产品化分发

目标：

- 让 Worker 从“脚本启动的服务”演进成“可迁移的算力客户端”

#### Task P3-1：定义 Worker 客户端架构

任务说明：

- 明确 Worker 客户端化的分层与边界

推荐三层结构：

1. 启动层：`Worker Launcher`
2. 运行层：现有 Worker 服务内核
3. 资源层：模型、配置、日志、缓存、FFmpeg、运行时资源

详细子任务：

- 输出 launcher 与 worker 的边界设计
- 定义本地配置文件位置
- 定义模型缓存位置
- 定义日志与状态目录
- 设计升级策略

建议产出：

- `docs/worker-launcher-design.md`

验收标准：

- 客户端化基于现有 Worker 内核演进，不另起一套系统

#### Task P3-2：Worker Launcher MVP

任务说明：

- 做一个最小可用的启动器

启动器应具备：

- 环境检查
- GPU / CUDA 检查
- 端口检查
- Coordinator 连通性检查
- 配置引导
- 启动 / 停止 Worker
- 基础日志查看

技术方向可选：

- `PyInstaller`
- `Nuitka`
- `embedded python + launcher`

详细子任务：

- 选择技术路线
- 实现首次启动向导
- 实现启动前环境检测
- 实现启动 Worker 进程
- 实现启动失败提示

验收标准：

- 非开发者可以较低门槛在新电脑上拉起 Worker

#### Task P3-3：模型管理器

任务说明：

- 把模型管理从手工目录操作升级为可控流程

详细子任务：

- 检查模型是否存在
- 提供按需下载
- 校验模型完整性
- 管理缓存目录
- 清理旧模型
- 切换默认模型

验收标准：

- 模型不再依赖手动拷贝和手动整理目录

#### Task P3-4：客户端运行体验

任务说明：

- 把 Worker 客户端做成适合长期常驻的节点应用

详细子任务：

- 托盘化或最小化常驻
- 开机自启
- 当前状态展示
- 最近任务摘要
- 最近错误提示
- 一键打开日志目录
- 一键测试 Coordinator 连接

验收标准：

- Worker 更像一个“节点应用”而不是纯命令行服务

### P4：多 Worker 与调度中心

目标：

- 从单 Worker 架构演进到真正的多节点系统

#### Task P4-1：多 Worker 注册与发现

详细子任务：

- 设计 Worker 注册协议

## 10. 结论

这份 roadmap 的核心思想是：

- 先稳定当前单 Worker 主链路
- 再补工程化和可维护能力
- 再补最符合自用场景的轻量自动化能力
- 再推进 Worker 客户端化
- 最后再扩展到多 Worker 和真正的分布式调度

如果按这条路线推进，SSUBB 会从“能跑的个人原型”逐步演进成“可长期依赖的家庭媒体字幕基础设施”。
